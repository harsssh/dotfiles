# CLAUDE.md ユーザー設定

## 全般

- 日本語で回答する

## 文章

- 記号は全角ではなく半角を使う
- 記号、英字の周りに 1 つスペースを入れる
- 太字や斜体などの装飾はしない。ただし、見出しや箇条書きなど文構造を表現する記法は使ってよい。
- 順序に意味がない列挙は順序リストではなく箇条書きを使う。
- 調査結果を報告するときは、最後にまとめを書く

## ツール

- ファイル検索には組み込みの Glob ツール、Bash にフォールバックする場合は `fd` を使う
- テキスト検索には組み込みの Grep ツール、Bash にフォールバックする場合は `rg` を使う
- alias で `rm` は interactive に実行されるので、Claude がファイルを削除するときは `rm -f` を使う
- git-switch, git-restore を使って安全に git リポジトリを操作して。git-checkout, git-reset を乱用しない。

## コーディング

### 設計・品質

- コードには How, テストコードには What, コミットログには Why, コードコメントには Why not を書く
- 関数の処理内容を説明するだけのコメントは書いてはいけない。説明が必要なほど長い関数は分割を検討する。
- 境界で変換、中では型を信頼する: レイヤ境界で入力を正規化し、内側は型付きオブジェクトで扱う
- Tell-Don't-Ask: オブジェクトに処理を依頼する
- 同じロジックを複数レイヤに置かない
- mutable より immutable な表現を使う
- 計算と副作用を混ぜず、pure / impure の境界を明確にする
- formatter, linter のエラーを ignore コメントで無効化してはいけない

### 命名

- 述語は肯定形で命名
- 仕様書の語彙をコードに反映する。独自の語を作らず、仕様・コード・テスト・API で同じ概念を同じ語で呼ぶ
- 英文法的に自然な識別子を選ぶ

### テスト

- テストデータや期待値がドメイン概念・既存データに紐づくなら、その定数・変数を参照する式で書く

## 言語別

### Nix

- 作業が完了したら `nix flake check` で検証する
- 再現性を最重視して設定を書く
- すべてを無理に Nix で管理しなくて良い。手軽に Nix の恩恵を受けれる部分だけ Nix 化すること。
- warning は無視せず可能な限り解消すること

### Ruby

- 3.1+ の hash shorthand を活用する
- 重複する setup は shared_examples / shared_context に括り出す
